It is likely that your logos and favicons are hosted on Kinde, but you may also manage externally hosted assets. Here's how assets are referenced in your custom code.

## Favicons

The Kinde infrastructure package ships with helper methods for accessing these:

```jsx
import {
	getFallbackFaviconUrl,
  getSVGFaviconUrl,
} from "@kinde/infrastructure";
```

### Usage

```jsx
<link rel="icon" href={getFallbackFaviconUrl()} sizes="48x48">
<link rel="icon" href={getSVGFaviconUrl()} type="image/svg+xml" />
```

## Logos

The Kinde infrastructure package ships with helper methods for accessing these:

```jsx
import {
	getLogoUrl,
  getDarkModeLogoUrl,
} from "@kinde/infrastructure";
```

### Usage

```jsx
// Always use light logo
<img
	src={getLogoUrl()}
	alt={event.context.widget.content.logoAlt}
/>

// Always use dark logo
<img
	src={getDarkModeLogoUrl()}
	alt={event.context.widget.content.logoAlt}
/>

// User preference
<picture>
	<source
		media="(prefers-color-scheme: dark)"
		srcset={getDarkModeLogoUrl()}
	>
	<img
		src={getLogoUrl()}
		alt={event.context.widget.content.logoAlt}
	/>
</picture>
```

## Org logos

Both functions accept the org code as an argument to return the relevant organization logo url:

```jsx
// Org logos
// Pass org code in
const {orgCode} = event.request.authUrlParams
<img
	src={getLogoUrl(orgCode)}
	alt={event.context.widget.content.logoAlt}
/>
```
# External Assets in Kinde

## Overview
When developing with Kinde, you'll likely need to integrate static assets like stylesheets, fonts, and images. This guide explains how to effectively manage external assets while maintaining Kinde's strict Content Security Policy (CSP).

## Content Security Policy (CSP) Considerations
Kinde implements a robust security policy that restricts how external resources can be loaded. 

### Domain-Based Asset Hosting
- **Key Requirement**: Assets must be hosted on a subdomain that shares the domain of your custom domain
- **Important Note**: You cannot use the subdomain of the custom domain itself for asset hosting

**Example Configurations**:
- If your custom domain is `account.myapp.com`:
  - Valid asset subdomains: 
    - `assets.myapp.com`
    - `static.myapp.com`
    - `cdn.myapp.com`
- If your custom domain is `auth.example.com`:
  - Valid asset subdomains:
    - `assets.example.com`
    - `static.example.com`

### Asset Hosting Best Practices

#### CSS Files
- **Hosting Options**:
  - Self-host stylesheets on a compatible subdomain
  - Minify CSS for optimal performance
  - Use compressed file formats

**Optimization Tips**:
- Use tools like `cssnano` or `clean-css` for minification
- Consider using CSS preprocessors with built-in optimization

#### Custom Fonts
- **Recommended Font Format**: `.woff2`
- **Benefits of `.woff2`**:
  - Smallest file size
  - Broad browser support
  - Faster loading times

**Font Optimization Strategies**:
- Use font subsetting to reduce file size
- Implement `font-display: swap` for improved perceived performance
- Host fonts on a consistent subdomain

#### Images
- **Preferred Image Formats**: 
  - `.webp`
  - `.avif`
- **Image Optimization Techniques**:
  - Use dedicated image optimization tools
  - Compress images without significant quality loss
  - Implement responsive image techniques

**Recommended Optimization Tools**:
- `ImageOptim`
- `squoosh.app`
- `sharp` (Node.js library)

## Implementation Example

```html
<!-- Correct asset loading -->
<link rel="stylesheet" href="https://assets.example.com/styles/main.min.css">
<style>
  @font-face {
    font-family: 'CustomFont';
    src: url('https://static.example.com/fonts/customfont.woff2') format('woff2');
    font-display: swap;
  }
</style>
```
